.env 內容一覽：共 5 個欄位，前 3 個是固定值（手動填），後 2 個由 setup.py 自動換取並寫入。

欄位類型來源有效期
FB_APP_ID固定FB 開發者後台永久
FB_APP_SECRET固定FB 開發者後台永久（可 reset）
FB_PAGE_ID固定FB 粉專關於頁永久
FB_LONG_LIVED_USER_TOKEN動態setup.py 換取60 天
FB_PAGE_ACCESS_TOKEN動態setup.py 換取不過期 *

* 必須由 long-lived user token 取得時才不過期；若由 short-lived 取得會跟著 user token 過期。

1

前置條件

執行 setup.py 前需要準備好的東西

環境檢查清單

  • 有一個 Facebook 個人帳號，是要操作的粉專的 管理員
  • 該 FB 帳號已在 Meta for Developers 註冊為開發者
  • 本機有 python3（3.7+）
  • 已 clone 本 repo，並切到專案目錄
2

建立 FB App

在 Meta for Developers 後台建立 App，取得 APP_ID / APP_SECRET / PAGE_ID

建立 App 步驟

  1. 進入 My Apps → 點選 developers.facebook.com/apps → Create App

  2. App type 選 Business

  3. 填 App 名稱與聯絡 email（例如 esg-hermes）

  4. 在 Settings → Basic 取得： App ID 與 App Secret（按 Show）

📌

FB_PAGE_ID 怎麼拿：到要操作的 FB 粉專 → 點 關於 (About) → 最下面有 粉專編號 (Page ID)。

3

設定權限

hermes agent 需要的 FB API 權限與 App Review 狀態

需要的權限清單

權限用途需 App Review
pages_show_list列出可管理的粉專否（開發階段）
pages_read_engagement讀粉專互動資料正式上線需要
pages_manage_posts發文 / 編輯貼文正式上線需要
pages_read_user_content讀粉專留言 / 訊息正式上線需要
⚠️

開發 / 測試階段：App 還沒提交 App Review 之前，只有 App 的 Admins / Developers / Testers 名單裡的 FB 帳號可以拿到這些權限。請先到 App Roles 把測試人員加進去。

4

取得 Short-Lived User Token

在 Graph API Explorer 抓一張 1~2 小時的暫時 token，馬上拿去跑 setup.py

Graph Explorer 操作步驟

  1. 進入 Graph API Explorer

  2. 右上 Meta App 切換到剛建立的 App

  3. User or Page 維持 User Token

  4. Permissions 加入上一步表格中的權限

  5. 按 Generate Access Token → 跳轉 FB 同意視窗 → 允許

  6. 畫面上方的 Access Token 欄位就是 Short-Lived User Token（有效期僅 1~2 小時）

🚨

不要把 Short-Lived Token 貼進 git commit：雖然 1~2 小時就會過期，但被洩漏的期間仍有 API 操作風險。

5

執行 setup.py

一鍵：short-lived → long-lived → page token → 寫入 .env

互動模式（推薦）

cd .claude/skills/facebook-page-setup
python3 scripts/setup.py

第一次跑會問你 4 個值，之後再跑會自動帶入既有 .env，只要貼新的 short-lived token 即可。範例輸出：

==================================================
 FB App .env 設定工具
==================================================
FB_APP_ID: <your_app_id>
FB_APP_SECRET [****...****]: (按 Enter 沿用)
FB_PAGE_ID [<your_page_id>]: (按 Enter 沿用)

請從 Graph API Explorer 取得 Short-Lived User Access Token
  https://developers.facebook.com/tools/explorer/
  選擇 App ID = <your_app_id>
  權限: pages_show_list, pages_read_engagement, pages_manage_posts

Short-Lived User Token: <short_lived_token>

[1/2] 換取 Long-Lived User Access Token ...
      OK  (****...****)
[2/2] 取得 Page Access Token ...
      OK  (****...****)

.env 已寫入：~/.hermes/.env（權限 600，僅本人可讀）

非互動模式（CI / Script 用）

python3 scripts/setup.py \
  --app-id <your_app_id> \
  --app-secret <your_app_secret> \
  --page-id <your_page_id> \
  --token <short_lived_token>

setup.py 內部做了什麼

  1. 呼叫 /oauth/access_token with grant_type=fb_exchange_token → 把 short-lived 換成 long-lived user token

  2. 計算 appsecret_proof = HMAC-SHA256(access_token, app_secret)

  3. 呼叫 /me/accounts（帶上 long-lived token + appsecret_proof）→ 取得該帳號所有可管理粉專

  4. 從清單找出符合 PAGE_ID 的 page → 取出該 page 的 access_token

  5. 把 5 個欄位合併寫入 ~/.hermes/.env（保留既有其他 key，不覆蓋整檔；檔案權限設為 600）

6

部署到 server

把 .env 放到 hermes agent 的工作目錄

產出的 .env 範例

FB_APP_ID=<your_app_id>
FB_APP_SECRET=<your_app_secret>
FB_PAGE_ID=<your_page_id>
FB_LONG_LIVED_USER_TOKEN=<long_lived_token>
FB_PAGE_ACCESS_TOKEN=<page_access_token>
FB_GRAPH_VERSION=v25.0

SCP 直接送上 server

scp .env user@your-hermes-server:/path/to/hermes/agent/
✅

正式環境建議：用 secrets manager（AWS SSM / Vault / Doppler）注入這幾個變數，不要把 .env 留在 server 檔案系統上。

7

Token 更新

long-lived user token 每 60 天要重抓一次
Token有效期更新方式
FB_LONG_LIVED_USER_TOKEN60 天重跑 setup.py + 新的 short-lived token
FB_PAGE_ACCESS_TOKEN不過期 *同上（自動連帶更新）

* 雖然不過期，但若 user 改密碼 / 取消授權 / App 被 reset secret，就會失效。建議每 30~45 天主動更新一次。

8

排雷 FAQ

setup.py 報錯時對照

setup.py 報 (#100) Invalid OAuth access token

  • Short-lived token 已過期（> 1~2 小時）→ 重抓
  • token 是從錯的 App 抓的 → 確認 Graph Explorer 右上 App 設定
  • App Secret 填錯 → 對照 FB 後台

報 找不到 Page ID xxx

  • 該 FB 帳號不是該粉專的管理員 → 到粉專設定加管理員
  • Page ID 填錯 → 再次到粉專 → 關於頁確認
  • Graph Explorer 抓 token 時忘了勾 pages_show_list → 重抓

拿到的 Page Token 一下子就過期

Page Token 的有效期會繼承來源 user token。一定要用 long-lived user token 去取 page token；setup.py 已經幫你處理這層，不要繞過去自己用 short-lived token 直接打 /me/accounts。

App Secret 不小心 push 到 GitHub 了

  1. 立刻到 FB 開發者後台 → Settings → Basic → Reset App Secret

  2. 用新的 secret 重跑 setup.py

  3. 更新 server 上的 .env

為什麼 setup.py 需要 appsecret_proof？

FB 的「Require app secret」設定開啟時（建議開），所有 API 呼叫都必須附上 appsecret_proof（HMAC-SHA256 of access_token with app_secret as key），證明請求方真的握有 App Secret。setup.py 內部已自動計算並附上。
